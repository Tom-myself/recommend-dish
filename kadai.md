# 「レシピの森」開発における主要な課題と解決策（ES作成用資料）

このドキュメントは、就職活動のエントリーシート（ES）や面接で「学生時代に最も苦労したこと」「技術的な困難をどう乗り越えたか」を説明するための素材として、これまでの開発の軌跡をまとめたものです。

ESの基本構成である **STAR法（状況・課題・行動・結果）** に沿って書きやすいように構成しています。

---

## エピソード1：フレームワークの制約と「理想の設計」の衝突（アーキテクチャの改善）

**【技術キーワード】** Java `record` (不変性), MyBatis, レイヤードアーキテクチャ, 責務の分離

### 状況と課題 (Situation & Task)
最新のJavaの機能である `record` を用いて、データの状態が途中で変わらない「完全な不変オブジェクト（Immutable）」としてアプリケーションを設計したいと考えました。しかし、データベース通信を担う MyBatis の仕様（遅延ロードを行うための `@Many` アノテーション）が、コンストラクタのみで初期化を行う `record` と真っ向から衝突しました。
結果として、SQLの取得カラム数とオブジェクトの引数数が合致しないというマッピングエラーが多発し、システムがクラッシュする事態に直面しました。

### 行動 (Action)
最初は「不変性を諦めて、変更可能なクラス（Setterを持つクラス）に妥協する」という場当たり的な対応をしました。しかし、「これではバグを生みやすい脆弱な設計になってしまう」と思い直し、アーキテクチャの根本的な見直しを図りました。
原因を深掘りした結果、「DB層（Mapper）が直接、画面表示用の複雑なオブジェクト（DTO）を作ろうとしている」という**責務境界の曖昧さ**が原因であることに気づきました。
そこで、以下のレイヤー分離を行いました。
1. **DB層**: SQLの結果と完全に一致する「軽量なEntity」のみを返すように修正。
2. **ビジネスロジック層 (Service)**: DB層から受け取った軽量Entityに、関連データ（材料や手順など）を結合し、最終的な画面用の不変オブジェクト（`record`）を組み立てる役割に特化。

### 結果 (Result)
フレームワーク（MyBatis）の制約に負けることなく、アプリケーション全体でオブジェクトの不変性を維持することに成功しました。また、「MapperはDBを知り、Serviceは画面との橋渡しをする」という教科書通りの美しいレイヤードアーキテクチャを実現でき、今後の仕様変更にも強い堅牢なコードベースを構築できました。

## エピソード2：ステートレスな認証システム（JWT）の導入とセッション管理

**【技術キーワード】** JWT (JSON Web Token), Spring Security, React (Context API), ステートレス認証

### 状況と課題 (Situation & Task)
「レシピ生成」のような誰でも使える公開機能と、「お気に入り登録・料理記録」のようなログイン必須の非公開機能を両立させる必要がありました。また、将来的なサーバーの拡張性（スケールアウト）を考慮し、サーバー側にログイン状態を保持する旧来のセッション方式ではなく、JWT（JSON Web Token）を用いたステートレスな認証システムを構築することが課題でした。

### 行動 (Action)
バックエンド（Spring Boot）では、リクエストのたびに送信されるJWTを検証する専用のフィルター（`JwtAuthenticationFilter`）を実装し、公開APIと非公開APIのアクセス制御を厳密に行いました。
フロントエンド（React）では、通信のたびに認証ヘッダーにトークンを付与する仕組みを構築しました。特に苦労したのは「UX（ユーザー体験）の向上」です。トークンの有効期限が切れた際、単にエラーを出すのではなく、フロントエンド側で期限を監視（タイマーを設定）し、自動的にセッション切れの通知画面へ誘導して再ログインを促す仕組みを実装しました。

### 結果 (Result)
サーバーのメモリを圧迫しない、モダンでセキュアな認証基盤が完成しました。また、トークン切れ時のシームレスなUI制御により、セキュリティを担保しつつもユーザーにストレスを与えないアプリケーションに仕上げることができました。

## エピソード3：クラウド環境へのデプロイと「コールドスタート」へのUX対策

**【技術キーワード】** Render, クラウドデプロイ, CORS, 非同期通信, UX（ユーザー体験）向上

### 状況と課題 (Situation & Task)
ローカル環境で完成したアプリを、無料クラウドサービス（Render）を用いて全世界に公開するフェーズでの課題です。フロントエンドとバックエンドを別々のサーバーで稼働させたため、ドメイン間の通信制限（CORS）エラーに直面しました。
また、最大の課題は「無料プラン特有のコールドスタート問題」でした。アクセスがないとバックエンドサーバーが休止状態になり、次にユーザーがアクセスした際、サーバーが起動するまでに数十秒の待機時間が発生してしまい、ユーザーが「アプリが壊れている」と誤解して離脱してしまうリスクがありました。

### 行動 (Action)
CORSの問題は、Spring Boot側のセキュリティ設定で本番環境のドメインを明示的に許可することで解決しました。
コールドスタート問題に対しては、技術的に起動時間を早めることは不可能だったため、「UXのアプローチ」で解決を図りました。React側で、画面表示と同時にバックエンドへ「死活監視（ヘルスチェック）」の通信を5秒間隔で行う仕組み（ポーリング）を実装しました。サーバーが起きていない間は「シェフが準備をしています...」といった遊び心のあるアニメーションバナーを表示し続け、起動が完了した瞬間にフェードアウトさせる処理を組み込みました。

### 結果 (Result)
本番環境での安定稼働を実現しました。サーバーの起動遅延というインフラ側の弱点を、フロントエンドのUI/UXの工夫によってカバーし、ユーザーの離脱を防ぐ洗練されたプロダクトに仕上げることができました。

## 💡 ESを書く際のアドバイス
これらのエピソードをESに書く際は、以下の流れを意識すると面接官に響きやすくなります。
1. **背景**: なぜその技術（recordやJWTなど）を使おうとしたのか（目的意識）
2. **困難**: その結果、どんな壁にぶつかったか（エラー内容や設計のジレンマ）
3. **解決策の思考プロセス**: 単に「ググって直した」ではなく、「なぜその設計が悪いのか根本原因を考え、アーキテクチャレベルで直した」「インフラの制限をフロントのUXでカバーした」という**あなたならではの工夫・思考**をアピールする。
