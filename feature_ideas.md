# 🚀 レシピの森 — 機能追加アイデア集

現在のアプリの技術スタック（React / Spring Boot / PostgreSQL / MyBatis / Docker）を活かしつつ、  
新しい技術を組み合わせてアプリを進化させるアイデアをまとめました。

各案ごとに **「新しく使う技術」「実装イメージ」「ESでのアピールポイント」** を記載しています。

---

## 🟢 難易度：低め（1〜2日で実装可能）

### 案A：写真で食材を認識 → レシピ提案（AI画像認識）

**新しい技術:** Gemini Vision API（マルチモーダルAI）

**概要:**  
冷蔵庫の中身をスマホで撮影すると、AIが画像から食材を自動認識し、それをもとにレシピを提案してくれる機能。
現在は食材をテキストで入力しているが、写真1枚で済むようになりUXが大幅に向上する。

**実装イメージ:**
- フロントエンドに画像アップロード用のUIを追加（ドラッグ&ドロップ or カメラ撮影）
- バックエンドで画像をBase64エンコードし、Gemini APIに「この画像に写っている食材を列挙してください」とプロンプトを送信
- 返ってきた食材リストをそのまま既存のレシピ生成APIに渡す

**ESアピール:**
> テキスト入力だけでなく、マルチモーダルAI（画像認識）を活用して、「写真を撮るだけでレシピが生成される」直感的なUXを実現しました。

---

### 案B：レシピのSNSシェア機能（Open Graph + Web Share API）

**新しい技術:** OGP (Open Graph Protocol), Web Share API

**概要:**  
生成されたレシピをX（旧Twitter）やLINEにシェアできるボタンを追加。シェアリンクを開くと、レシピのタイトルや材料がOGP画像としてプレビュー表示される。

**実装イメージ:**
- バックエンドにレシピを一時共有するためのURLエンドポイント（`/api/recipe/share/{id}`）を作成
- OGPメタタグ（`og:title`, `og:description`, `og:image`）を動的に生成
- フロントエンドでは `navigator.share()` (Web Share API) を使ってネイティブのシェアダイアログを呼び出す

**ESアピール:**
> Web Share APIとOpen Graph Protocolを実装し、レシピをSNSで共有できるソーシャル機能を追加。リンクプレビューの動的生成にも取り組みました。

## 🟡 難易度：中くらい（3〜5日で実装可能）

### 案C：1週間の献立を自動プランニング（献立カレンダー）

**新しい技術:** FullCalendar.js（高機能カレンダーライブラリ）, ドラッグ&ドロップ

**概要:**  
「今週の予算は5000円」「月曜は魚、水曜は肉」といった条件を入力すると、AIが1週間分の献立を一括で生成し、カレンダーにマッピングしてくれる機能。
さらに、気に入らない日はドラッグ&ドロップで入れ替えたり、個別に再生成できる。

**実装イメージ:**
- 専用ページ `MealPlanPage.jsx` を新規作成
- バックエンドに「1週間分のレシピを一括生成する」エンドポイントを追加（Gemini APIへのプロンプトを工夫）
- 生成結果を `meal_plans` テーブル（`user_id`, `plan_date`, `recipe_id`）に保存
- フロントエンドでは FullCalendar.js を使ってドラッグ&ドロップ可能なインタラクティブなカレンダーを表示

**ESアピール:**
> AIによる1週間分の献立自動生成と、ドラッグ&ドロップで直感的に編集できるインタラクティブなカレンダーUIを実装しました。

### 案D：買い物リストの自動生成（お気に入りレシピから逆算）

**新しい技術:** PWA (Progressive Web App) + ブラウザ通知

**概要:**  
お気に入りに登録したレシピ、または献立カレンダーに入っているレシピの材料を自動で集計し、「今週の買い物リスト」を生成する機能。
PWA化することで、ホーム画面に追加してスーパーでオフラインでもリストを確認できるようにする。

**実装イメージ:**
- `shopping_lists` テーブル（`user_id`, `ingredient_name`, `is_checked`）を新規作成
- お気に入りレシピの材料を集約し、重複を排除して1つのリストにまとめるService層のロジックを実装
- フロントエンドはチェックリスト形式のUIで、タップで「買った」を記録
- `manifest.json` と Service Worker を追加してPWA化 → スマホのホーム画面からネイティブアプリのように起動

**ESアピール:**
> PWA (Progressive Web App) に対応させ、スマホにインストール可能なネイティブアプリ風の体験を実現。Service Workerによるオフラインキャッシュも実装しました。

### 案E：GitHub Actions による CI/CD パイプライン構築

**新しい技術:** GitHub Actions, Docker Hub (または GitHub Container Registry)

**概要:**  
`main` ブランチにプッシュしたら自動でテスト → ビルド → デプロイまで行われるCI/CDパイプラインを構築する。
コードの品質保証とデプロイの自動化により、プロフェッショナルな開発フローを実現する。

**実装イメージ:**
- `.github/workflows/ci.yml` を作成
- ワークフロー内容：
  1. `mvn test` でバックエンドの単体テストを実行
  2. `npm run build` でフロントエンドのビルドを確認
  3. テスト成功時のみ Render へ自動デプロイ（Render の Deploy Hook URL を利用）
- バックエンドに JUnit5 のテストクラスを追加（Service層のロジックテスト）

**ESアピール:**
> GitHub Actionsを用いたCI/CDパイプラインを構築し、テスト → ビルド → デプロイの自動化を実現。継続的な品質担保を行える開発体制を整備しました。

## 🔴 難易度：高め（1週間以上・挑戦的）

### 案F：レシピの「味の傾向」分析ダッシュボード（データ可視化）

**新しい技術:** Recharts (グラフライブラリ), データ分析ロジック

**概要:**  
これまでの料理ログやお気に入りデータを分析し、「あなたは和食が多め」「揚げ物率が高い」「野菜が少ない傾向」といった食生活の傾向をグラフで可視化するダッシュボード機能。

**実装イメージ:**
- 専用ページ `DashboardPage.jsx` を新規作成
- バックエンドに集計APIを追加（例：`/api/stats/taste-profile`）
  - AIに過去のレシピ傾向を分析させ、カテゴリ分類（和食/洋食/中華、肉/魚/野菜など）を行う
- フロントエンドではRecharts を使ってレーダーチャートや棒グラフで結果を表示
- 「もっと野菜を食べたほうがいいかも？」といったAIからのアドバイスも表示

**ESアピール:**
> ユーザーの食生活データをAIで分析し、Rechartsを用いたインタラクティブなダッシュボードで食生活の傾向を可視化。データドリブンな健康管理機能を実装しました。

### 案G：WebSocket によるリアルタイム「料理中タイマー」

**新しい技術:** WebSocket (Spring WebSocket + SockJS), リアルタイム通信

**概要:**  
レシピの手順ごとにタイマーをセットでき、複数のタイマーを同時に管理できる「クッキングタイマー」機能。
WebSocketを使うことで、ブラウザのタブを閉じてもサーバー側でタイマーが動き続け、完了時にプッシュ通知を送る。

**実装イメージ:**
- Spring WebSocket を導入し、`/ws/timer` エンドポイントを作成
- フロントエンドは SockJS + STOMP でWebSocket接続し、タイマーの開始/停止/完了をリアルタイムで同期
- Notification API を使ってブラウザ通知を送信（「パスタの茹で時間が終わりました！」）
- 複数タイマーを並行管理するUIをReactで構築

**ESアピール:**
> WebSocket（双方向リアルタイム通信）を活用し、サーバーサイドでタイマーを管理するクッキングタイマー機能を実装。HTTPのリクエスト/レスポンスモデルとの違いを理解した上で適材適所の技術選定を行いました。

---

### 案H：OAuth 2.0 によるGoogleログイン

**新しい技術:** OAuth 2.0, Spring Security OAuth2 Client, Googleアカウント連携

**概要:**  
「Googleでログイン」ボタンを追加し、Googleアカウントで簡単にサインアップ/ログインできるようにする。
既存のJWT認証と共存させ、メールアドレス/パスワード方式とOAuth方式を選べるようにする。

**実装イメージ:**
- Google Cloud Consoleで OAuth 2.0 クライアントIDを取得
- Spring Security に `spring-boot-starter-oauth2-client` を追加
- 認証フローの追加：Google → 認可コード取得 → バックエンドでトークン交換 → JWTを発行
- `users` テーブルに `auth_provider` カラムを追加（`LOCAL` or `GOOGLE`）
- フロントエンドに「Googleでログイン」ボタンを配置

**ESアピール:**
> OAuth 2.0プロトコルを用いたGoogleアカウント連携を実装。既存のJWT認証との共存設計を行い、複数の認証方式を統合するセキュリティアーキテクチャを構築しました。

## 💡 個人的なおすすめ

**まずは「A. 写真で食材認識」から始めるのが最もおすすめです。**  
理由：
1. 既にGemini APIを使っているので、Vision機能の追加は比較的スムーズ
2. 「写真を撮るだけでレシピが出る」というインパクトが圧倒的に強い
3. ESや面接で「マルチモーダルAI」というキーワードが使える
4. 実装量が少なく、短期間で目に見える成果が出る

その次に **「E. CI/CD」** を入れると、「機能開発だけでなく、開発プロセス自体を改善する視点も持っている」というアピールになり、技術者としての評価が格段に上がります。
